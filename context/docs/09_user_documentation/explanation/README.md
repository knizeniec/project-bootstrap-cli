# Explanation Overview

> **Purpose**: explain when to write understanding-oriented documentation and route readers to conceptual background pages.
> **Audience**: end users who need a mental model, rationale, or tradeoff discussion rather than a direct procedure.
> **When to update**: update when explanation-writing guidance or conceptual navigation changes.

## How to use this template

Use this page when deciding whether content should help users understand ideas instead of complete a task. Explanation pages should answer “why?” and “what should I understand?” rather than “what do I do next?”

- Write explanation when context and reasoning help users make better decisions.
- Keep examples illustrative, not procedural.
- Link to tutorials, how-to guides, and reference pages for action, lookup, or practice.

## When to write an explanation

Write an explanation when the reader needs background, framing, or tradeoffs to use the product well. In Diátaxis terms, explanation is understanding-oriented and gives meaning to the rest of the documentation set.

- Good fit: permission model, workflow rationale, architectural limitation, data model concept.
- Bad fit: first-run lesson, exact field list, or step-by-step task instructions.

## Explanation characteristics

Good explanation pages introduce the concept, describe why it matters, compare alternatives, and point to further reading. They can be a little more discursive than other modes, but they should still stay structured and easy to scan.

- Include tradeoffs and boundaries so users understand when the concept applies.
- Use short examples to make the concept concrete without turning the page into a how-to.
