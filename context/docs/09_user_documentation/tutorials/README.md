# Tutorials Overview

> **Purpose**: explain when to write tutorial content and route readers to guided learning-by-doing documents.
> **Audience**: end users who are new to the product or learning a workflow for the first time.
> **When to update**: update when tutorial-writing guidance, onboarding flow, or tutorial navigation changes.

## How to use this template

Use this page when deciding whether a new page should teach through a guided exercise instead of solving a narrow task. Tutorials should be safe to follow step by step, with no missing moves for the learner to invent.

- Write a tutorial when the reader needs confidence and context, not just an answer.
- Prefer one clear path from start to finish over branching options.
- Link out to how-to or reference pages instead of overloading the lesson.

## When to write a tutorial

Write a tutorial when the reader is learning by doing and benefits from a complete, hand-held sequence. In Diátaxis terms, tutorials are for study and practice rather than for fast problem-solving.

- Good fit: first login, first project setup, first successful workflow.
- Bad fit: fixing one error, checking a parameter, or debating design tradeoffs.

## Tutorial characteristics

Good tutorials are linear, success-oriented, and explicit about what the learner should observe after each step. They should not assume prior product intuition beyond the listed prerequisites.

- Include concrete commands, clicks, or inputs in the order they must be done.
- Add a verification checkpoint so the learner knows they are still on track.
