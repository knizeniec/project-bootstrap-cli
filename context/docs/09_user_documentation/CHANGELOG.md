# Changelog

All notable changes to the end-user documentation area should be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and versions should be linked once real releases exist.

> **Purpose**: provide a rolling, user-facing summary of notable documentation and product-surface changes across releases.
> **Audience**: end users who want a quick history of what changed without reading every individual release note.
> **When to update**: update whenever a user-visible documentation or product change should be recorded in the next release summary.

## How to use this template

Keep entries short, grouped by release, and written from the user's point of view. Update `Unreleased` as changes land, then move those bullets into a dated or versioned section when the release ships.

- Use the standard Keep a Changelog subheadings shown below.
- Omit empty subsections in published release entries if they add noise.
- Link to the detailed release note for the release when one exists.
