# How-To Guides Overview

> **Purpose**: explain when to write problem-oriented how-to guides and route readers to practical task instructions.
> **Audience**: end users who already know the product basics and need help completing a specific task.
> **When to update**: update when task-guide conventions or navigation for problem-oriented content changes.

## How to use this template

Use this page when deciding whether content should help a user complete one concrete task. How-to guides should assume the reader has intent, not curiosity: they came to get something done.

- Write a how-to when the title can start with an action verb.
- Keep only the steps needed for the task at hand.
- Link to tutorials for onboarding and to reference pages for exact syntax or field meanings.

## When to write a how-to

Write a how-to when a reader asks, “How do I do this?” and needs a reliable path to a result. In Diátaxis terms, how-to guides are practical and problem-oriented, not educational or encyclopedic.

- Good fit: export a report, reset a setting, invite a user, recover from a common failure.
- Bad fit: first-time teaching, full concept overview, or complete API listing.

## How-to characteristics

Good how-to guides state the goal up front, list only the prerequisites that matter, and finish with a clear verification step. They should be easy to scan during real work and short enough to use under time pressure.

- Prefer imperative steps with exact labels, commands, or navigation paths.
- Add troubleshooting only for common blockers that materially stop task completion.
